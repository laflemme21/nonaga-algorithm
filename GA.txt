Genetic Algorithm Architecture Specification

Objective: Build a modular, highly parallelized Genetic Algorithm (GA) framework in Python suitable for High-Performance Computing (HPC) environments (specifically SLURM-based clusters). The framework must allow for easy comparison of different GA operators (selection, crossover, mutation) and parallel architectures (Master-Slave, Coarse-Grained Island Models).

1. Core Architectural Paradigm

The system will strictly adhere to the Strategy Design Pattern. The main GA orchestrator class will rely on Dependency Injection, accepting Abstract Base Classes (ABCs) for all genetic and parallel operations. This prevents hardcoding and allows hot-swapping of methods.

Required Abstract Base Classes (ABCs):

SelectionStrategy: Interface for choosing parents.

CrossoverStrategy: Interface for reproduction.

MutationStrategy: Interface for genetic alteration.

FitnessFunction: Decoupled interface for the objective function to allow isolated execution on remote worker nodes.

ParallelBackend: Interface handling the distribution of tasks or populations.

2. Confirmed Specifications

A. Genome Representation

Type: 1D array (list) of integers.

Length: Currently 8 parameters, but the architecture MUST support variable/dynamic lengths determined at initialization.

Data Structure: Use NumPy 1D arrays for the individuals to allow for fast, vectorized crossover and mutation operations.

B. HPC Parallel Framework (Master-Slave & Island)

Use the concurrent.futures.Executor interface design pattern.

Rationale: This allows the GA to be developed locally using Python's built-in ProcessPoolExecutor, and seamlessly transition to research-grade mpi4py.futures.MPIPoolExecutor or dask.distributed on the HPC without altering the GA's core logic.

C. Tracking & Logging

Output format: CSV.

The GA should append generation-level metrics (e.g., Generation ID, Best Fitness, Average Fitness, Worst Fitness) to a CSV file to avoid keeping massive logs in memory and to ensure data is saved if an HPC job times out.

3. Implementation Roadmap

Phase 1: Minimum Viable Product (Simple GA) -> CURRENT FOCUS

Implement the base framework with the ABCs.

Create one simple concrete implementation for each strategy (e.g., Random Selection, Single-Point Crossover, Random Int Mutation).

Get a simple Master-Slave parallelization working where the population is evaluated in parallel using concurrent.futures.

Phase 2: Operator Expansion (Future)

Add complex strategies (Tournament, Uniform Crossover, etc.) utilizing the established ABCs.

Phase 3: Coarse-Grained Architecture (Future)

Implement the Island Model topology and migration policies.
